The move: Save Branch as Scene

Right now the Wolf is a node built by hand inside main.tscn. It has children (Sprite2D, CollisionShape2D, SwingTimer, AggroArea, HealthBar) and a script. To get three wolves, you could duplicate the node — but then a bug fix requires touching all three by hand.

Instead, we apply the Lesson 14 instancing move to the wolf itself: Save Branch as Scene extracts it into wolf.tscn. From that point on you drag the file into the world as many times as you want — every copy gets all the children and the full script automatically. Fix the scene once; every instance benefits.

Save Branch as Scene

Right-click any node in the Scene dock → Save Branch as Scene. The node and all its children become a standalone .tscn file. In Main it now shows as an instance (the film-strip icon).

Instance

A live copy of a scene placed in another scene. Change the source file and every instance updates. Give each instance a different position in the Inspector without touching the source file.

Self-contained

The wolf was already safe to extract because nothing in its script points outside itself. It reaches its own children with $SwingTimer / $HealthBar — relative paths that survive being moved into a new file.

Part A — extract the wolf into its own scene

1

Open main.tscn. In the Scene dock, right-click the Wolf node → Save Branch as Scene.

2

Save it as res://wolf.tscn. The Wolf in Main's Scene dock gains a film-strip icon — it's now an instance of the file.

3

Ctrl+S to save main.tscn. Press F5 — the game should play exactly as before. One wolf, same behaviour. The extraction was invisible to the player.

◈ why the extraction is safe

The wolf's script accesses $SwingTimer, $HealthBar, $Sprite2D — all relative paths within its own branch. It detects the player via is_in_group("player"), not a hardcoded node path. Nothing pointed out at the parent scene — that's the property that makes a branch safe to extract.

Part B — drag in the pack

1

In the FileSystem dock, find wolf.tscn. Drag it into the Main viewport twice. Godot names the new instances Wolf2 and Wolf3.

2

In the Inspector (with each selected), adjust Position so the three wolves are spread out — they shouldn't all spawn on top of each other.

3

You can also set a different level on each wolf in its Inspector, since level is an @export — per-instance overrides work on any exported variable.

▲ try it now — see the gap

Press F5 and walk in. All three wolves chase and bite you — each has its own independent brain, for free. But press attack: you can only ever damage the one named Wolf, because start_auto_attack() still uses get_node_or_null("../Wolf") — a hardcoded path that can only ever find the first one. Wolf2 and Wolf3 are effectively unkillable.

The recurring lesson of this codebase

"Stop pointing at one named node" is a theme you'll see again and again. A hardcoded path is a contract: "there will always be exactly one node with this exact name in this exact place." The moment that contract breaks — a second wolf, a renamed node, a scene reorganisation — the code silently fails.

The fix is a targeting system: instead of the player hunting for a node by name, the player waits to be told who the target is — via a mouse click. That's Lesson 19.

◇ the WoW model

In WoW your auto-attack is single-target: it hits whoever you've targeted. You pick a target by clicking. Switch targets and your swings follow without resetting the rhythm. Hitting everything at once is a separate ability — Whirlwind — which you'll build later in the combat arc.
