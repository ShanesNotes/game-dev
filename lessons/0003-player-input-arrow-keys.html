The new idea: if

Last lesson the robot always moved. A controllable character only moves when a key is held. That word "when" is an if statement: run some lines only if a condition is true.

GDScript · shape of an if
if <something is true>:
    # these indented lines run only then

Same indentation rule as a function: the indented lines belong to the if.

How Godot reads the keyboard

Godot has built-in input actions already mapped to the arrow keys: "ui_up", "ui_down", "ui_left", "ui_right". You ask "is this action being pressed right now?" with:

GDScript
Input.is_action_pressed("ui_right")

That gives back true (held) or false (not held) — exactly what an if wants. (Godot Docs: Input examples)

One more tool: Vector2

Position has two parts: x (left↔right) and y (up↔down). A Vector2 bundles those two numbers into one value, written (x, y). We'll build a direction as a Vector2 — holding right makes (1, 0), up-and-right makes (1, -1).

⚠ y is upside-down

In 2D, y grows downward, so "up" is -1, not +1.

The obvious way — and its hidden flaw

Four separate ifs feel natural. Here's what that looks like:

GDScript · the four-if approach
var direction = Vector2.ZERO
if Input.is_action_pressed("ui_right"): direction.x += 1
if Input.is_action_pressed("ui_left"):  direction.x -= 1
if Input.is_action_pressed("ui_down"):  direction.y += 1
if Input.is_action_pressed("ui_up"):    direction.y -= 1

It works — mostly. Hold right and down together and the direction is (1, 1). Mathematically, that vector is ~1.41× longer than a straight direction. The character runs about 41% faster on diagonals. That's a real bug — and it's hard to notice until someone speedruns your game.

◈ the diagonal-speed trap

A straight direction like (1, 0) has length 1. A diagonal like (1, 1) has length √2 ≈ 1.41 — so multiplying it by speed gives ~41% more velocity. The fix is normalizing: shrink any vector back to length 1 before using it. The four-if approach forces you to do this manually every time.

The better way: Input.get_vector()

Godot has a single call that returns a ready-made direction Vector2 — and it fixes the diagonal bug automatically, among other things:

Normalized

The returned vector is always length ≤ 1, so diagonals are no faster than straights. You never have to think about it.

Analog

A gamepad stick reports how far it's pushed, not just on/off. get_vector preserves that, so a gentle tilt gives a gentle speed.

Deadzone handling

Sticks drift slightly at rest; get_vector ignores those tiny inputs so the character doesn't wander on its own.

◈ controller support, for free

Godot's built-in ui_left/right/up/down actions come pre-bound to a gamepad's left stick and D-pad. Once movement reads them via get_vector, plug in a controller and the left stick already works — no Input Map changes needed.

The code

Replace your whole script with this.

GDScript · player.gd
extends Sprite2D

var speed = 200

func _process(delta):
    var direction = Input.get_vector("ui_left", "ui_right", "ui_up", "ui_down")
    position += direction * speed * delta
Input.get_vector("ui_left", "ui_right", "ui_up", "ui_down") — order is left, right, up, down (negative-x, positive-x, negative-y, positive-y). Returns a direction Vector2, normalized, already handling diagonals and analog sticks.
position += direction * speed * delta — one line for all directions. Nothing held → direction is (0, 0) → robot stays put. Delta keeps it frame-rate independent.
