1 · The attack table as a deep module

Strong
player.gd:111–202 reference/wow-combat-values.md

Problem

The combat formulas — miss_chance, dodge_chance, parry_chance, glancing_chance, crit_chance, glancing_damage, the one-roll band-walk in attack(), and the three rage-conversion functions — are pure math (TELOS law 4: real numbers, cited from wow-combat-values.md). But they live interleaved with node calls inside the Player module. The interface to “how combat resolves” is currently all of player.gd: to trust the band ordering (roll < miss + dodge + parry + …) you must read the implementation. That ordering is exactly where this class of bug hides, and today it is untestable — attack() cannot run without a scene tree, a HUD, and a live wolf.

Solution

Extract a CombatTable module (combat_table.gd, class_name CombatTable, static functions, zero node references): CombatTable.roll(attacker, defender, rng) → Outcome where Outcome is {kind: MISS|DODGE|PARRY|GLANCING|CRIT|HIT, damage: int}, plus CombatTable.rage_from_dealing(...) / rage_from_taking(...). The Player’s attack() shrinks to: roll the table, then present the outcome (floating text, sparks, shake). The WoW citation moves to the top of the new file.

Benefits

  • Leverage: one two-function interface hides ~90 lines of researched formulas and the band-walk invariant.
  • Locality: every future combat change (L27 XP scaling, L33 Whirlwind per the roadmap) lands in one cited file.
  • The interface is the test surface: the repo gets its first headless test seam — godot --headless --script tests/test_combat_table.gd asserting exact values against the reference doc (e.g. “+3 levels ⇒ parry 14%”). A seeded RNG makes the band-walk deterministic.
  • Deletion test: passes. Delete CombatTable and seven formulas + an ordering invariant scatter back into Player. It concentrates complexity; it doesn’t move it.
  • Teachability: the attack table becomes one artifact a lesson can point at whole — and lesson 27 will extend exactly this code. Trigger: fired (combat arc L27–33 is the declared next step).

2 · Player announces, HUD listens — a signal seam

Strong
player.gd:18,21 hud.gd:16–27 wolf.gd

Problem

The Player module’s interface currently includes the HUD’s internal layout: get_node("../../HUD/UnitFrame/HealthBar"). The seam leaks both ways — the HUD module polls player.target and reads target.health every frame. This fragility already bit: when Player moved under World during the visual overhaul, the hard paths broke and had to be hand-patched. Worse, the curriculum itself teaches the fix — TELOS law 5 celebrates “events over polling (L11)” — so the exhibit currently contradicts its own doctrine.

Solution

Player emits health_changed(value, max), rage_changed(value), target_changed(target); Wolf emits health_changed. The HUD module connects in _ready() and its _process() polling disappears. Player loses all knowledge that a HUD exists.

Benefits

  • Deletion test (run as a thought experiment on the HUD): today, deleting the HUD scene crashes Player in _ready(). After: the game runs headless without UI — which is also what makes combat simulation tests possible.
  • Locality: HUD layout changes (and the L27/28 XP bar the roadmap implies) never touch player.gd again.
  • Leverage: any future listener — nameplates, portraits, an XP bar, a sound module — subscribes to the same four signals.
  • Teachability: this is a lesson, not just a refactor: “wire the HUD the way we wired aggro in L11.” Trigger: fired twice (paths already broke once; L11 doctrine already taught).

3 · One CombatFX module for floating text & bursts

Worth exploring
player.gd:140,204,230 wolf.gd:70,101

Problem

Five near-identical presentation spawners exist across two modules: spawn_text_over, spawn_number ×2, spawn_sparks, spawn_blood. Each is a shallow helper (interface ≈ implementation) and each bakes in a scene-tree assumption (get_parent().add_child(...)) — the same assumption class that broke during restructuring. Changing how damage numbers look means visiting three call sites in two files.

Solution

One CombatFX module (autoload): CombatFX.number(at, amount, opts), CombatFX.text(at, s, color), CombatFX.burst(at, color, count) — spawning into a dedicated FX layer instead of whatever get_parent() happens to be.

Benefits

  • Deletion test: passes, modestly. Five helpers collapse into one adapter behind a three-function interface.
  • Locality: crit styling, fonts, rise curves — one file. Y-sort/z-index quirks of world-space labels get one home.
  • Natural companion to candidate 2 (could be the same lesson’s second half), or folded in when combat FX next changes. Smaller win on its own — hence the badge.

4 · ForestPlan: split the plan from the planting

Worth exploring
forest_generator.gd (298 lines)

Problem

The forest generator is already deep from the outside (drop it on a TileMapLayer, set four exports) — but internally, pure planning math (path winding, dirt mask, dual-grid corner bits, scatter and grove rules) is fused with node construction and three hard scene paths (Road, World, Player). Invariants like “props never sit on dirt” or “road half-width stays within 0.7–1.7” can only be checked by booting the full game and looking.

Solution

ForestPlan.build(seed, width, height) → plan — pure data: dirt mask, ground tiles, road cells with their transition bits, prop placements (type + position), spawn cells. The node script becomes a thin applier that walks the plan. Two adapters of the same plan are plausible later (apply-to-scene, draw-to-debug-image), which would make this a real seam rather than a hypothetical one.

Benefits

  • Testability: placement invariants become headless assertions over data; seed regressions become diffs of plans, not screenshots.
  • AI-navigability: the plan is inspectable — “why is there a rock on the road?” becomes a data question.
  • Law-5 caution (why not Strong): the trigger hasn’t fully fired. Recent forest bugs were wiring, not planning logic, and there is only one consumer of the plan today. Park it until the next real generation bug or a second consumer (minimap? lesson visual?) appears.

5 · A contract between lessons and code (drift check)

Strong
lessons/*.html (26) lessons/index.html projects/first-steps/*

Problem (actively on fire)

Lessons anchor to the code through prose: copy-paste snippets, node names, asset filenames, input-action names. There is no machine-checkable link — so yesterday’s visual overhaul silently contradicted at least four lessons: L03 teaches ui_left/ui_right… (code now uses move_*); L13/15/16/25 teach get_node("../HUD/HealthBar") (now ../../HUD/UnitFrame/HealthBar); L05 slices tileset.png (now terrain.png + a Road layer); L17 drags sword.png (scene now uses sword_held.png). The exhibit’s public promise — the course is the codebase’s build order — is breakable by any refactor, and breakage is discovered by a learner mid-lesson.

Solution

Give each lesson a tiny claims manifest (front-matter JSON or a sibling .claims.json): the input actions, node paths, asset files, and named constants the lesson’s snippets rely on. One check_lessons.py validates claims against project.godot, the scenes, and assets/ — runnable locally or in CI. Drift becomes a failing check naming the lesson and the claim.

Benefits

  • Locality: the lesson↔code contract — currently smeared across 26 HTML files and one human memory — gets one home and one checker.
  • Leverage: every future refactor (including AI-driven ones like yesterday’s) gets an immediate, named list of curriculum repairs.
  • Telos guard: this is exhibit tooling, so it must stay small (anti-telos: “polishing the exhibit instead of learning”). A claims file is ~10 lines per lesson; the checker is one script. It protects the public promise without becoming a project.
  • Trigger: fired — four concrete drifts exist right now and need hand-repair anyway; writing the claims is the repair inventory.

Top recommendation

Start with №1, the CombatTable — with №5 as the urgent chore beside it.

№1 is the purest depth win available: it passes the deletion test decisively, it gives this repo its first headless test seam (there are currently zero tests anywhere), it puts the cited WoW numbers (law 4) in one file, and — decisive under this telos — Lesson 27 (XP & leveling) will extend exactly this code. Doing the extraction as the opening move of the combat arc means every formula added in L27–33 lands behind a tested interface, and the extraction itself is lesson-sized (“one lesson, one win”).

№5 is not a depth refactor but it protects the exhibit’s public promise, and its trigger isn’t just fired — it’s burning: four lessons are contradicted by the code today. The claims inventory doubles as the repair list. №2 follows naturally as the lesson after (it’s L11 doctrine applied to the HUD, and the XP bar will want those signals).
